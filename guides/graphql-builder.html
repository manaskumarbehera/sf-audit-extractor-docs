What It Does

The GraphQL Builder lets you compose and execute Salesforce GraphQL queries directly from TrackForcePro. Write queries with syntax highlighting and smart formatting, define variables for reusable queries, and explore formatted responses with error highlighting.

Salesforce GraphQL (via the UI API) is a powerful alternative to SOQL for fetching record data and metadata. Use it to retrieve related records efficiently in a single request, or to build more flexible queries than SOQL allows.

How to Access

Open the TrackForcePro workspace and click the GraphQL tab. You'll see the query editor on the left and the response panel on the right (initially empty until you run a query).

Query Editor

The left panel is a full-featured CodeMirror 6 editor built specifically for GraphQL:

  • Syntax Highlighting — GraphQL keywords, fields, and types are color-coded for readability.
  • Auto-Indentation — The editor automatically indents nested structures.
  • Bracket Matching — Hovering over or selecting a bracket highlights its pair.
  • Line Numbers — Line numbers on the left help you reference query sections.
  • Dark Theme Support — The editor adapts to your theme preference (light/dark).

Paste or type your Salesforce GraphQL query directly into the editor. The query must follow Salesforce GraphQL schema.

Example query structure:
query GetAccounts { uiapi { query { Account { edges { node { Id Name BillingCity } } } } } }

Variables Panel

The variables panel (right side of the query editor) accepts JSON that defines reusable variables for your query. Variables allow you to parameterize queries so you can reuse the same query structure with different data.

  • JSON Format — Enter variables as valid JSON (e.g., { "accountId": "001xx000..." }).
  • Reference in Query — Use $variableName in your GraphQL query to reference a variable.
  • Reusable Queries — Keep the query template the same and swap variables for different inputs.

If your query doesn't use variables, leave the variables panel empty. Variables are optional.

Example:
Variables: { "searchTerm": "Acme" }
Query: query SearchAccounts($term: String) { uiapi { query { Account(where: { Name: { like: $term } }) { ... } } } }

Executing Queries

Once your query is ready:

  • Click the Run button (usually at the top of the editor), or
  • Use the keyboard shortcut (typically Ctrl+Enter or Cmd+Enter).

TrackForcePro sends your query and variables to Salesforce's GraphQL endpoint. The response appears in the response panel on the right.

Response Explorer

The response panel displays the result of your GraphQL query in several formats:

  • Formatted JSON — Your response is pretty-printed with syntax coloring and proper indentation for readability.
  • Error Highlighting — If the query fails, errors appear with clear messages explaining what went wrong (e.g., schema validation errors, field not found).
  • Collapsible Sections — Nested objects and arrays can be collapsed/expanded to focus on relevant data.
  • Status Info — The panel shows the HTTP status code and execution time.

You can copy response sections or export the entire response for further analysis.

Common GraphQL Errors:
  • Cannot query field "xxx" on type "yyy" — The field doesn't exist on that object. Check the Salesforce GraphQL schema.
  • Expected Name, found "}" — Syntax error in your query. Check brackets and commas.
  • Invalid variable — A variable reference is missing a definition in the variables panel.

Tips & Best Practices

  • Start Simple — Salesforce GraphQL has specific schema rules. Build a simple query first (e.g., fetch 5 Account records), verify it works, then expand to complex nested queries.
  • UI API Pattern — Most Salesforce GraphQL queries follow the uiapi pattern. If you're unfamiliar, look for Salesforce GraphQL documentation or examples in your org's GraphQL endpoint schema explorer.
  • Use Variables for Reusability — If you'll run similar queries multiple times with different parameters, define variables. This keeps your query template clean.
  • Test on Small Data Sets — Add WHERE clauses or LIMIT equivalents to test queries before fetching large result sets.
  • GraphQL vs. SOQL — GraphQL is stronger for fetching related records in a single request. SOQL is better for simple record queries. Use both when appropriate.
  • Response Size — Large GraphQL responses can take longer to render. If your response is very large, try narrowing the field selection or adding filters.
💡 Pro Tip: Save your successful query templates as notes or in a text editor. GraphQL schema is consistent within your org — reusing proven templates saves time and reduces syntax errors.
