🌐 REST Explorer Guide

What It Does

The REST Explorer gives you a direct interface to Salesforce's REST APIs without needing Postman, Insomnia, or terminal tools. Choose an HTTP method (GET, POST, PATCH, PUT, DELETE), enter an endpoint, set headers if needed, compose a request body, and execute. Responses appear formatted and syntax-highlighted for easy reading.

Perfect for exploring APIs, testing endpoints, debugging integrations, and learning Salesforce's REST schema.

How to Access

Open the TrackForcePro workspace and click the REST Explorer tab. The request builder and response viewer are ready to use.

Request Builder

The request builder has four main sections:

Method Selector

Choose your HTTP method from the dropdown:

  • GET — Retrieve data. No request body needed.
  • POST — Create a new record or trigger an action. Requires a request body (JSON).
  • PATCH — Update an existing record. Include the fields to update in the body.
  • PUT — Replace an entire record. Requires a full request body.
  • DELETE — Delete a record. Usually no request body.

Endpoint URL

Enter your endpoint relative to your Salesforce instance. TrackForcePro automatically prepends your org's instance URL and appends your access token, so you only need the path.

Examples:
/services/data/vXX.0/sobjects/
/services/data/vXX.0/sobjects/Account/describe
/services/data/vXX.0/sobjects/Account/001xx000...
/services/data/vXX.0/query/?q=SELECT+...

Headers (Optional)

Add custom headers if needed. Key-value pairs like:

Content-Type: application/json
Accept: application/json

In most cases, TrackForcePro sets these automatically. Add custom headers for specialized endpoints.

Request Body

For POST, PATCH, and PUT requests, enter your request body as JSON. For example, to create an Account:

{ "Name": "New Account Inc.", "BillingCity": "San Francisco", "BillingCountry": "USA" }

GET and DELETE requests typically don't need a body.

Common Endpoints

Here are frequently-used Salesforce REST API endpoints to get you started:

Endpoint Method Purpose
/services/data/vXX.0/sobjects/ GET List all available SObjects (objects) in your org.
/services/data/vXX.0/sobjects/Account/describe GET Get detailed metadata for a specific object (fields, relationships, etc.).
/services/data/vXX.0/sobjects/Account GET Retrieve recent Account records.
/services/data/vXX.0/sobjects/Account/001xx000... GET Fetch a single Account record by ID.
/services/data/vXX.0/sobjects/Account/001xx000... PATCH Update an Account record.
/services/data/vXX.0/sobjects/Account POST Create a new Account record.
/services/data/vXX.0/sobjects/Account/001xx000... DELETE Delete an Account record.
/services/data/vXX.0/query/?q=SELECT+Id,Name+FROM+Account GET Execute a SOQL query (URL-encoded).
/services/data/vXX.0/tooling/query/?q=SELECT+... GET Execute a Tooling API query (for metadata).
/services/data/vXX.0/limits/ GET Check org limits (API calls, storage, etc.).

Note: Replace vXX.0 with your API version (e.g., v60.0 for Winter 2024).

Response Viewer

After you send a request, the response panel displays:

  • Status Code — HTTP status (200 OK, 400 Bad Request, 404 Not Found, etc.).
  • Response Headers — Metadata about the response (content type, rate limit info, etc.).
  • Response Body — The actual data, formatted as JSON with syntax highlighting and proper indentation.
  • Collapsible Sections — Nested objects can be expanded/collapsed for easier navigation.

If the request fails, error details appear with a clear explanation of what went wrong.

Common HTTP Status Codes:
  • 200 OK — Request succeeded.
  • 201 Created — Resource created successfully.
  • 400 Bad Request — Invalid request format or missing required fields.
  • 401 Unauthorized — Authentication failed (token expired or invalid).
  • 403 Forbidden — You lack permission for this action.
  • 404 Not Found — Resource doesn't exist (wrong ID or endpoint).
  • 500 Server Error — Salesforce server error. Try again later.

Tips & Best Practices

  • Auto-Authentication — TrackForcePro automatically includes your org's instance URL and access token. No manual auth setup needed.
  • Test on Sandboxes First — Use a sandbox org to test PATCH/PUT/POST/DELETE operations before running them on production.
  • Explore with GET First — Start with GET requests to understand the API response structure before attempting writes.
  • Check Rate Limits — Call /services/data/vXX.0/limits/ to see your remaining API call quota.
  • Format JSON Carefully — Invalid JSON in the request body will cause a 400 error. Use a JSON validator if unsure.
  • Save Successful Requests — Copy working endpoints and request bodies to a notes file for quick reference later.
  • Use Describe Endpoints — Call /sobjects/{Object}/describe to see field names, types, and validation rules before creating/updating records.
  • Bookmark Common Endpoints — Query limits, org info, and version info are endpoints you'll reuse frequently.
💡 Pro Tip: Keep your instance URL and API version (v60.0, v61.0, etc.) handy. The REST Explorer is fast once you memorize a few key endpoints. You can prototype integrations, test before writing Apex, or debug API calls in real time.
