🔴 1. Current UX Problems

Three core issues identified through structural analysis of the extension's UI.

Problem 1 Data Explorer is overloaded (5 sub-tabs + 2 advanced)

Data Explorer currently houses 7 distinct tools that serve completely different purposes: record lookups, org management, user administration, security auditing, favicon customization, metadata browsing, and automation inspection. Users must navigate 2–3 levels deep to reach frequently-used tools like Custom Labels or Login As.

Current: 3-level depth to reach Custom Labels
Tab Bar:   🔍 Audit | 📡 Events | 📢 LMS | 📊 SOQL | 🔗 GraphQL | 💾 Data Explorer | ⚙️
                                                                    │
Sub-tab:  Record Scanner | Org & Favicon | User Mgr | 🔐 Security | 🏢 Org Tools
                                                                    │
Panel:    Login As | Custom Labels | Deployments | Jobs | Debug Logs | Limits | Utils
                         ↑ 3 clicks deep!
Problem 2 Org & Favicon is duplicated across two locations

The Favicon Customizer appears in two places: (1) Data Explorer → "Org & Favicon" sub-tab (full editor with org list, live/draft previews, environment presets) and (2) Settings → "Org & Favicon" sub-tab (simplified version with org info card and a basic favicon form). Users don't know which one to use, and the two don't share state consistently.

Problem 3 Settings doesn't show saved orgs or per-org actions

Despite storing rich org data (orgRecords, favicons, environment types, notes, pinned records, query history), the Settings tab doesn't surface any of this. Users can't see which orgs they've visited, can't manage saved data, and can't access per-org preferences from Settings — they have to navigate to Data Explorer → Org & Favicon to do any org management.

⚠️ Trade-offs & Risks
  • 📏Tab bar width: 8 labeled tabs + 3 icon tabs may feel crowded at smaller popup widths. Mitigation: Use shorter labels ("Records" not "Record Scanner", "Org Tools" stays compact). At narrow widths, tab text already collapses to icons-only.
  • 🔗Existing bookmarks & shortcuts: Users who bookmarked data-tab → org-tools subtab need redirect logic. Mitigation: Add backward-compatible aliases in popup.js tab routing.
  • 🧪Test updates: ~15-20 test files reference "data" tab or Data Explorer sub-tabs. These need updating for new tab IDs.
  • 📌Sidebar compatibility: Existing user-saved sidebar shortcuts using old __TFP_SUBTAB__org-tools patterns need migration. Mitigation: Auto-migrate in settings_helper.js on first load.
⚡ Breaking change: The data-tab="data" tab ID changes to data-tab="records". The old data ID should be kept as an alias for one version cycle (v1.9.0) before removal in v2.0.0.
